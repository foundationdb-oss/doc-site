## Why Teams Choose FoundationDB

<div class="feature-grid" markdown>

<div class="feature-card featured" markdown>
<div class="feature-icon">:material-shield-check:</div>

### True ACID at Any Scale

Multi-key, multi-range transactions with **serializable isolation**—the strongest guarantee possible. No eventual consistency surprises. No distributed transaction headaches. Just correctness, always.

[Learn about transactions →](concepts/transactions.md)
</div>

<div class="feature-card featured" markdown>
<div class="feature-icon">:material-flask:</div>

### Simulation-Tested

Our deterministic simulation framework tests **millions of failure scenarios** before every release. Network partitions, disk failures, process crashes—we've already found and fixed the bugs that would hit you in production.

[How simulation testing works →](architecture/simulation-testing.md)
</div>

<div class="feature-card" markdown>
<div class="feature-icon">:material-server-network:</div>

### Distributed by Design

Scales horizontally across commodity servers. Add nodes to increase capacity without application changes. Built for clusters from 3 nodes to hundreds.

</div>

<div class="feature-card" markdown>
<div class="feature-icon">:material-lightning-bolt:</div>

### High Performance

Millions of operations per second with single-digit millisecond latencies. Optimized for both read and write-heavy workloads.

</div>

<div class="feature-card" markdown>
<div class="feature-icon">:material-shield-refresh:</div>

### Fault Tolerant

Automatic failover and recovery. Your data survives hardware failures, network partitions, and even datacenter outages.

</div>

<div class="feature-card" markdown>
<div class="feature-icon">:material-layers-triple:</div>

### Multi-Model Foundation

Build documents, graphs, tables, or queues on top of the ordered key-value store. The layer concept lets you create any data model.

</div>

</div>
